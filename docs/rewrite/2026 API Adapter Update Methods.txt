API Adapter Updates Project 2026
Project Overview
This project aims to bring the PIC-SURE API adapters up to date with the existing PIC-SURE user interface and data dictionary. Specifically, the project scope includes:
* Deduplicating data dictionary output fields[a]
* Using the updated data dictionary
* Using the new version of the query object (v3 for advanced filtering)
* Enable user to combine AND and OR filtering with grouping[b][c][d]
* Removing consents from the adapter, as these will be handled by the backend (as part of the advanced filtering project)


Overview of User Actions
There are two main paths of user interactions with the PIC-SURE API. The first is the user that starts in the user interface and then moves to the API. The second is the user that does all of their work in the API. [e]


UI to API
1. User starts in PIC-SURE UI 
2. User creates their cohort and saves dataset ID
3. User then starts up an analysis workspace and desires to bring their data in using a dataset ID


Based on this, the key functionality that is needed is the:
* Ability to connect to the API
* Ability to bring in participant-level data via a dataset ID
* Ability to print previous dataset IDs


API Only
1. User starts in analysis workspace with PIC-SURE API
2. User searches the data dictionary
3. User creates their cohort
4. User bring data into analysis workspace


Based on this, the key functionality that is needed is the:
* Ability to connect to the API
* Ability to search the data dictionary
   * Facets
   * Metadata[f]
* Ability to initialize a query
* Ability to add filters to a query
   * Filter
   * Select
   * Any record[g]
   * Require
* Ability to see their query[h]
* Ability to export data from a query 
* Ability to save and access dataset IDs similar to the UI


Defining Functions
To define functions for developers, we will use the following fields to provide context, descriptions, and requirements. 
1. User goals and workflows
Describe what scientists are trying to do, in plain language:
* Primary workflows (step‑by‑step):
   * “Search variables in the data dictionary, inspect them, then choose a subset.”
   * “Build a cohort with filters, then export participant‑level data to a DataFrame.”
2. Function list and behavior
For each function you want, provide:
Name and purpose in 1–2 sentences
* e.g. “XYZ_function lets a scientist do … and returns ...”
* Inputs (arguments):
   * Name
   * Type in simple terms: “string”, “list of strings”, “True/False flag”
   * Whether it’s required or optional
   * What it means in user terms
(e.g. “text – words to search for in variable name, label, or description.”)
   * Outputs / return value:
   * What kind of object: “list of variable objects”, “Pandas DataFrame”
   * What a “variable object” contains, in simple fields: id, name, label, units, etc.
   * Examples:
   * Minimal working examples in Python, exactly how you expect users to use it.
3. Help / docstrings for each function
For each function, write a short help text the engineers can turn into a docstring:
   * One‑line summary
   * Description of key parameters, especially anything that might confuse users
   * Example usage
For example (you can give it in plain text; engineers can convert to proper docstring):
   * Function: function_name
   * Description: Overview of what the function does
   * Parameters:
   * param1 (string, optional): words to search for in variable name, label, or description
   * Returns:
   * What it returns, including type and expected fields
4. Expectations about errors and edge cases
Tell the team how you want the API to behave when things go wrong or are empty:
   * What should happen when:
   * Search finds no variables (e.g. “return empty list / empty DataFrame, not an error”).
   * A query has invalid filters (e.g. “raise a clear, user‑friendly error message”).
   * A user passes a wrong type (string instead of list): “Error should explain what they did wrong and how to fix it.”
   * Any guardrails, for example:
   * Max number of participants or variables for one export.
   * Timeouts
   * Size limits
5. Performance and usability expectations
Non‑technical but important guidance[i]:
   * “Search should feel responsive for up to X variables.”[j]
   * “Exporting participant‑level data may take longer; show progress or at least be documented as potentially slow.”
   * “Default limits should be safe for use in notebooks (won’t overload memory).”
6. Backwards compatibility and versioning
Since you’re changing the API:
   * List the old functions and how they worked (at a high level).
   * Note which ones are:
   * Staying the same
   * Being renamed
   * Being removed or changed
   * Specify:
   * Whether you need backwards compatibility (e.g. old code should still run with warnings).
   * How you want deprecation communicated (warnings, docs, changelog).
7. Testing and examples
Clarify what you expect for validation and education:
   * Types of examples you want in docs, e.g.:
   * “Search then select variables, then build a query and export a DataFrame.”
   * “Small end‑to‑end example with fake data.”
   * Any acceptance criteria:
   * “A new user can copy‑paste a search example and get results in a DataFrame within 5 minutes.” 
   * Included in Gitbooks / R in-line documentation
